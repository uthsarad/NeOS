# Strategic Directive

## Phase 1 — Product Alignment Check
- **Product Vision:** NeOS aims to be a highly predictable, rolling-release Arch Linux-based desktop operating system delivering a refined, Windows-familiar user experience right out of the box. Our fundamental goal is to provide a seamless transition to a Linux environment, combining Arch's flexibility with workstation reliability.
- **Alignment:** The product continues to align with its core vision. Recent improvements such as the offline install support via `tools/gen-install-repo.sh`, the redesigned Plymouth splash (animated cat), and the shift to Wayland/X11 autologin fixes indicate strong progress. However, our forward momentum is currently halted by critical unresolved validation tasks stemming from our systemd infrastructure hardening initiatives, specifically in `profile/airootfs/etc/systemd/system/neos-autoupdate.service` and `profile/airootfs/etc/systemd/system/neos-liveuser-setup.service`. The strict systemd sandboxing must be thoroughly verified by our specialists before we can safely resume standard feature development. We must prevent unverified baseline configurations from becoming entrenched technical debt.
- **Leverage:** The most critical action right now is to rigorously validate the systemd sandboxing configurations (`ProtectSystem=strict`, `ProtectHome=yes`, `NoNewPrivileges=yes`) across mission-critical services like `neos-autoupdate.service`. Sentinel has recently mitigated a symlink traversal vulnerability in `neos-liveuser-setup` but failed to clear the primary systemd validation debt. Furthermore, `neos-liveuser-setup.service` must remain unequivocally unsandboxed because it performs system-level account creation (`useradd -m`); we must ensure no regression occurs that would break live autologin by modifying `/etc` and `/home/liveuser`. We must aggressively clear this validation debt to re-establish a stable, reliable foundation for future roadmap milestones.

## Phase 2 — Technical Posture Review
- **Stability:** We have deployed aggressive security enhancements, applying strict sandboxing to core system services (e.g. `neos-autoupdate.service`). While this significantly hardens the system against potential exploits, it also introduces substantial architectural fragility due to strict read-only paths and reduced capabilities. We also recently added `tests/verify_service_hardening.sh` to enforce this posture, specifically ensuring `neos-liveuser-setup.service` is exempt to prevent silently breaking the live boot when attempting to write to user homes or global configurations.
- **Tech Debt:** Our validation debt is now at a critical threshold. The specialist task manifests for Sentinel (privilege auditing and `ReadWritePaths` verification) and Palette (logging clarity and UX) are currently pending execution. We are operating blind until these audits confirm the stability of the sandboxed services. Crucially, `neos-autoupdate.service` requires complex explicit `ReadWritePaths=/usr /boot /etc /var /opt /srv` and `CAP_SYS_MODULE` (for DKMS kernel module builds) that Sentinel must review to avoid breaking silent background package manager updates.
- **Overbuilding:** Implementing any new features or optimizations at this juncture would be technically irresponsible. Adding complexity on top of unverified infrastructure hardening exponentially increases regression risks and significantly complicates any subsequent debugging. We must definitively secure the base before building upward. Feature freeze is enacted.

## Phase 4 — Controlled Scope Definition
- **Impacted Files:** None.

No new code, bug fixes, refactoring, or superficial updates are permitted today. This strict lock must remain until specialists confirm the systemd configuration is robust.

## Phase 5 — Delegation Strategy
- **Architect:** Immediate and complete cessation of all coding activities. You must maintain this strict strategic pause without exception. Focus exclusively on system analysis and architectural planning for future phases, but execute absolutely no code modifications until the validation debt is fully cleared by the specialists. No bypasses for minor typos or superficial changes.
- **Bolt:** Maintain the current performance baseline without introducing any changes. Optimization tasks are suspended until architectural stability is confirmed. No subshell optimizations or script rewriting.
- **Palette:** URGENT: Execute the pending logging UX validation task immediately. Rigorously verify that the strictly sandboxed services (`neos-autoupdate.service`) provide exceptionally clear, actionable error diagnostics. Ensure that any failures caused by read-only filesystem restrictions are explicitly logged in `journalctl` and not masked by generic exit codes (e.g., exit code 1 or 127). This is paramount for future maintainability.
- **Sentinel:** URGENT: Execute the pending privilege auditing task immediately. Conduct a comprehensive security review of the systemd sandboxing in `profile/airootfs/etc/systemd/system/neos-autoupdate.service`. Definitively validate that `ReadWritePaths` are correctly tailored to allow essential operations (e.g., access to `/var`, `/boot`, `/opt`, `/srv`, and `CAP_SYS_MODULE` for dkms) while preventing unauthorized access to other directories. Verify that `profile/airootfs/etc/systemd/system/neos-liveuser-setup.service` correctly remains unsandboxed to allow `/home/liveuser` creation and `useradd` modifications. This validation is non-negotiable for system security and operational integrity.
